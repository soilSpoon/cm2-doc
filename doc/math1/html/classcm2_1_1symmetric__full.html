Detailed Description

template<class T>
class cm2::symmetric_full< T >

Template symmetric full matrix, resizable, with ref-counting.

Template type T can be a POD (unsigned, double, char...) or a user-defined class (with visible default constructor, copy constructor, destructor and assignment operator).

The data are stored contiguously in memory, in a column oriented way (column-major order).

   [0] [1] [3] [6]
       [2] [4] [7]
           [5] [8]
               [9]

Just like dense1D from which it inherits many features, symmetric_full uses a "smart pointer" scheme to enable different matrices to share the same data and to manage proper memory deallocation.
Copy constructors and operators do not actually copy the data but copy the view to the data.
This is called shallow copy (vs hard copy).
When a symmetric_full is copied to another one, the copy is inexpensive and they both share the same data.
A change of an element in one matrix is viewed from the other one.

Warning
symmetric_full, as all variable-size containers, are not fully thread-safe because some resources (heap memory) can be shared between several threads.
Copies, assignments and reads are thread-safe, but not writes!
Note
symmetric_full uses a smart pointer with reference counting to manage memory deallocation.

TAny type with default constructor, copy constructor and assignment operator.

◆ reserve() [1/2]

template<class T>
bool cm2::symmetric_full< T >::reserve ( size_type n)
inline

Manual memory reservation without dimension change.

The main reason for using reserve() is efficiency.
If you know the capacity to which your matrix must eventually grow, then it is usually more efficient to allocate that memory all at once rather than resizing it several times.

The other reason for using reserve() is that you can control the invalidation of iterators.

Returns
True for success.
Note
reserve() has no effect for a value lesser than the current size.

Changes the dimension of the matrix (uninitialized).

Elements under min(dim(), n) are left unchanged.
If m > dim(), new rows and columns between dim() and n are uninitialized.
A reallocation occurs and iterators are invalided when n*(n+1)/2 > capacity().

Returns
True for success.
Note
When it is not necessary to keep the data, it is a good practice to clear the matrix before any resizing.

The capacity of the vector (always greater than or equal to size()).

Automatic reallocation occurs when size() reaches capacity().
It is then usually increased by a factor of two.

◆ reserve() [2/2]

template<class T, int VEC_ALIGNMENT = CM2_ALIGNMENT>
bool cm2::dense1D< T, VEC_ALIGNMENT >::reserve ( size_type n,
size_t growth_rate = 1 )
inlineinherited

Manual memory reservation without size change.

The main reason for using reserve() is efficiency.
If you know the capacity to which your vector must eventually grow, then it is usually more efficient to allocate that memory all at once rather than resizing it several times.

The other reason for using reserve() is that you can control the invalidation of iterators.

Returns
True for success.
Note
reserve() has no effect for a value lesser than the current size.

◆ resize() [3/3]

template<class T, int VEC_ALIGNMENT = CM2_ALIGNMENT>
bool cm2::dense1D< T, VEC_ALIGNMENT >::resize ( size_type n,
value_type v )
inlineinherited

Changes the size of the vector with initialization value for the new elements if any.

Elements under min(size(),n) are left unchanged.
If n > size(), new elements between size() and n are initialized to v.
If n > capacity(), a reallocation occurs and iterators are invalid.

Note
When the data are not to be kept, it is a good practice to clear the vector before any resizing.
Returns
True for success.
